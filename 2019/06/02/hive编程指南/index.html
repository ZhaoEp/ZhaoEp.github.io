《Hive编程指南》
2019-06-02

HiveQL

基本语句

LIMIT

用来限制返回的行数

列别名

新产生的结果列对原来的表来说是不存在的, 通常要给这些新产生的列起一个名称.

嵌套SELECT语句

对于嵌套查询, 别名是非常有用的.

CASE…WHEN…THEN 语句

与if条件语句类似, 用于处理单个列的结果查询.

LIKE 和 RLIKE

LIKE

其可以让我们通过字符串的开头或结尾以及指定特定的子字符串或当子字符串出现在字符串内的任何位置时进行匹配.

RLIKE

可以通过Java的正则表达式来指定匹配条件

什么情况下Hive可以避免进行MR

  • 本地模式

如果属性hive.exec.model.local.auto = true; Hive 会尝试使用本地模式执行其他操作
最后将该配置添加添加到$HOME/.hiverc配置文件中

  • WHERE 子句中过滤条件只是分区字段

关于浮点数比较

当用户写一个浮点数时, Hive会将该值保存为DOUBLE类型.

对于数字 0.2 来说, 其对于FLOAT类型是 0.2000001, 而对于DOUBLE类型是 0.200000000001
当表中的FLOAT值通过Hive转换为DOUBLE值时, 其产生的DOUBLE值是 0.200000100000, 这个值实际要比 0.200000000001 大.

和钱相关的都避免使用浮点数;
对于浮点数比较时, 要避免任何从窄类型隐式转换到更广泛类型的操作

GROUP BY

通常与聚合函数一起使用,按照一个或者多个列对结果进行分组,然后对每个组执行聚合操作

HAVING 语句

HAVING 子句允许用户通过一个简单的语法完成原本需要通过子查询才能对GROUP BY语句产生的分组进行条件过滤的任务.

如果没有使用HAVING子句, 需要使用嵌套SELECT子查询

JOIN

Hive中的JOIN只支持等值连接

当对3个或者更多个表进行JOIN连接时, 如果每个ON子句都使用相同的连接键的话, 那么只会产生一个MR任务.WHERE语句在连接操作执行后才会执行, 因此WHERE语句应该只用于过滤那些非NULL值的列值.

LEFT SEMI-JOIN

左半开连接会返回左边表的记录, 前提是其记录对于右边表满足ON语句中的判定条件.

SEMI-JOIN 通常比INNER JOIN更高效;
对于左表中一条指定的记录, 在右边表中一旦找到匹配的记录, Hive就会立即停止扫描.

map-join
  • set hive.auto.convert.join=true 在必要的时候使用map-join
  • set hive.mapjoin.smalltable.filesize=25000000 map-join时小表的大小

分桶表

  • set hive.optimize.bucketmapJOIN=ture 分桶表对应的桶map-join

对于桶键排序的分桶表, Hive可以执行一个更快的sort-merge JOIN, 需要设置的参数如下.

  • set hive.input.format=org.apache.hadooop.hive.ql.io.BucketizedHiveInputFormat;
  • set hive.optimize.bucketmapjoin=true;
  • set hive.optimize.bucketmapjoin.sortedmerge=true;

ORDER BY 和 SORT BY

ORDER BY

对查询结果进行全局排序,所有的数据都通过一个reducer进行处理.

SORT BY

局部排序, 只会在每个reducer中对数据进行排序, 即每个reducer中的数据是有序的, 但是并非全局有序.

因为ORDER BY操作可能会导致运行时间过长, 如果属性hive.mapred.mode的值是strict的话, 那么Hive要求这样的语句必须加有LIMIT语句进行限制.默认这个属性值是nonstrict

含有 SORT BY 的 DISTRIBUTE BY

DISTRIBUTE BY 控制 map 的输出在 reducer 中是如何划分的.

Hive要求DISTRIBUTE BY语句要写在SORT BY语句之前.

CLUSTER BY

CLUSTER BY 等于 DISTRIBUTE BY + SORT BY

使用DISTRIBUTE BY ... SORT BY或CLUSTER BY语句会剥夺SORT BY的并行性, 然而这样的可以实现输出文件的数据是全局排序的.

UNION ALL

可以将2个或多个表进行合并, 每一个union子查询都必须具有相同的列, 而且对应的每个字段的字段类型必须是一致的.

索引

一张表的索引数据存储在另外一张表中.

维护索引需要额外的存储空间同时创建索引也需要消耗资源

创建索引

对分区字段country建立索引

如果省略了 PARTITIONED BY , 那么索引表将会包含原始表的所有分区.

AS ... 语句指定了索引处理器, 也就是一个实现了索引接口的java类.

并不是一定要求索引处理器在一张新表中保留索引数据, 但是如果需要的话, 会使用到 IN TABLE.. 子句.

Bitmap 索引

bitmap索引普遍应用于排重后值较少的列.

重建索引

如果用户指定了 DEFERRED REBUILD, 那么新索引将呈现空白状态, 在任何时候, 都可以进行第一次索引创建或者使用ALTER INDEX对索引进行重建;

如果省略掉PARTITION, 那么将会对所有分区进行重建索引.

如果重建索引失败, 那么在重建开始之前, 索引将停留在之前的版本状态. 从这种意义上看,重建索引操作是原子性的.

显示索引

关键字FORMATTED是可选的. 增加这个关键字可以使输出中包含有列名称.
可以使用INDEXS替换INDEX, 这样输出中就可以列举多个索引信息了.

删除索引

如果有索引表的话, 删除一个索引将会删除这个索引表.

如果被索引的表被删除了, 那么其对应的索引和索引表也会被删除.
如果原始表的某个分区被删除了, 那么这个分区对应的分区索引也同时会被删除.

模式设计

按天划分的表

使用分区表.

关于分区

Hive 通常要对输入进行全盘扫描, 来满足查询条件.
通过创建分区可以优化一些查询, 但是同时可能会对其他一些重要的查询不利.

使用过多的分区, 会创建大量的非必须的Hadoop文件和文件夹. 一个分区就对应着一个包好有多个文件的文件夹. 如果指定的表存在数百个分区, 那么可能每天都会创建好几个万个文件. 如果保持这样的表很多年, 那么最终会超出NameNode对系统云数据信息的处理能力.
因为NameNode必须要将所有的系统文件的元数据信息保存在内存中.
虽然每个文件只需要少量字节大小的元数据(大约是150字节/文件), 但是这样也会限制一个HDFS实例所能管理的文件总数的上限.

MR会将一个job转换成多个task. 默认情况下, 每个task都是一个新的JVM实例, 都需要开启和销毁的开销. 对于小文件, 每个文件都会对应一个task.

在一些情况下, JVM开启和销毁的时间中销毁可能会比实际处理数据的时间消耗更长.

一个理想的分区方案不应该导致产生太多的分区和文件夹目录并且每个目录下的文件应该足够大, 应该是文件系统中块大小的若干倍.

同一份数据多种处理

对于每个表的分区

由于查询或者原始数据处理的某个步骤出现问题而导致需要对好几天的输入数据重跑ETL过程. 这时用户可能就需要执行那些一天执行一次的处理过程,来保证在所有的任务完成之前不会有job将临时表覆盖重写.

该方法的一个缺点是 : 用户将需要管理中间表并删除旧分区(这些任务很容易实现自动化处理)

分桶表存储数据

分区提供一个隔离数据和优化查询的便利的方式

并非所有的数据集都可以形成合理的分区.

分桶是将数据集分解成更容易管理的若干部分的另一个技术,
其会将用户指定的字段值进行Hash,分发到各个桶中.

向分桶表中插入数据

SET hive.enforce.bucketing = true;强制Hive为目标表的分桶初始化过程设置一个正确的reducer个数.如果没有使用该属性,那么我们就需要自己设置和分桶个数相匹配的reducer个数.
例如使用SET mapred.reduce.tasks=96,然后在INSERT语句中,需要在SELECT语句后增加CLUSTER BY语句.

优点 :

  • 因为桶的数量是固定的,所以它没有数据波动,桶对于抽样在合适不过.如果两个表都是按照user_id进行分桶的话,那么Hive可以创建一个逻辑上正确的抽样.
  • 分桶同时有利于执行高效的map-side JOIN.

为表增加列

Hive提供了SerDe抽象, 其用于从输入中提取数据, 输出数据.

SerDe 通常是非常宽松的.

例如, 如果某行的字段个数比预期的要少, 那么缺少的字段将返回null.
如果某行的字段个数比预期的要多, 那么多出的字段将会被省略掉.

通过这种方式, 无法在已有字段的开始或者中间增加新字段.

使用列存储表

使用列式存储的数据集特点

  • 重复数据
    • 假设有足够多的行, 像state和age字段这样的列将会有很多重复的数据.
  • 多列
    • 数据集中有非常多的字段, 而查询通常只会使用一个字段或者很少的一组字段.

总是使用压缩

压缩可以使磁盘上存储的数据量变小, 这样可以通过降低I/O来提高查询执行速度.

Hive可以无缝地使用很多压缩类型. 不适用压缩唯一令人信服的理由就是产生的数据用于外部系统或者非压缩格式是最兼容的.

但是压缩和解压缩都会消耗CPU资源,MR 任务往往是I/O密集型的,因此CPU开销通常不是问题.

不过,对于工作流这样的CPU密集型的场景,例如一些机器学习算法,压缩实际上可能会从更多必要的操作中获取宝贵的CPU资源,从而降低性能.

调优

使用EXPLAIN / EXPLAIN EXTENDED

限制调整

很多情况下LIMIT语句还是需要执行整个查询语句, 然后再返回部分结果的.

这种情况通常是浪费的, 所以应该尽可能地避免出现这种情况.

通过设置hive.limit.optimize.enable = true来开启,当使用LIMIT语句时,其可以对源数据进行抽样.

一旦hive.limit.optimize.enable = true,那么还会有两个参数可以控制这个操作,也就是hive.limit.row.max.size(某一行至少抽取的数量)和hive.limit.optimize.limit.file(最多能抽取的文件数);

缺点 : 有可能输入中有用的数据永远不会被处理到.

本地模式

当Hive输入的数据量非常小的时候,为查询触发执行任务的时间消耗可能会比实际job的执行时间要多的多.对于大多数这种情况,Hive可以通过本地模式在单台机器上处理所有的任务. 对于小数据集,执行时间可以明显被缩短.

用户可以通过设置属性 hive.exec.mode.local.auto=true, 来让Hive在适当的时候自动启动这个优化.

通常用户可以将这个配置写到$HOME/.hiverc文件中.

如果希望对所有的用户都使用这个配置, 那么可以将这个配置项增加到hive-site.xml文件中

并行执行

某个特定的job可能包含众多的阶段, 而这些阶段可能并非完全相互依赖的, 也就是说有些阶段是可以并行执行的, 这样可能使得整个job的执行时间缩短.

通过设置参数hive.exec.parallel = true,就可以开启并发执行.

在共享集群中. 需要注意下,如果job中并行执行的阶段增多,那么集群利用率就会增加.

严格模式

Hive提供了一个严格模式, 可以防止用户执行那些可能产生意想不到的不好的影响的查询.

通过设置hive.mapred.mode=strict可以禁止3种类型的查询.

  • 对于分区表,除非WHERE语句中含有分区字段过滤条件来限制数据范围,否则不允许执行.换句话说, 就是用户不允许扫描所有分区.

因为通常分区不都拥有非常大的数据集而且数据增加迅速. 没有进行分区限制的查询可能会消耗巨大的资源来处理该表.

  • 对于使用了ORDER BY语句的查询,要求必须使用LIMIT语句.

因为ORDER BY为了执行排序过程会将所有的结果数据分发到同一个reducer中进行处理,强制要求用户增加这个LIMIT语句可以防止reducer额外执行很长一段时间

  • 限制笛卡尔积的查询

调整mapper和reducer个数

确定最佳的mapper个数和reducer个数取决于多个变量. 例如输入数据量的大小以及对这些数据执行的操作类型等.

如果有太多的mapper和reducer任务,就会导致启动阶段,调度和运行job过程中产生过多的开销.
如果设置的数量太少,那么就可能没有充分利用好集群内在的并行性.

Hive是按照输入的数据量大小来确定reducer个数的.

hadoop dfs -count 文件路径可以查看某个文件的大小

hive.exec.reducers.bytes.per.reducer 默认值是1GB, 每个reducer接收的数据量.

默认值通常情况下比较合适.

对于某些情况,可以通过将reducer个数设置为固定的值,来进行验证.

Hive中默认的reducer个数为3个.

可以通过设置属性mapred.reducer.tasks的值为不同的值来确定最佳的reducer个数.

对于在共享集群上处理大任务时, 可以通过设置属性hive.exec.reducers.max阻止某个查询消耗太多的reducer资源.

JVM重用

JVM重用可以使得JVM实例在同一个job中重新使用N次.

N的值可以在Hadoop的mapred-site.xml文件中进行设置.

索引

索引可以用来加快含有GROUP BY语句的查询的计算速度.
